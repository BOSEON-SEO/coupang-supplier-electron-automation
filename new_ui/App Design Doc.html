Inbound Ops — 워크플로우 재설계

발주확정과 운송분배는 같은 차수 안에 묶여 있지만, 복합키 단위로는 서로 독립입니다. 이 분리 가능성을 활용하면 작업 페이스, 담당자, 화면 구조가 모두 달라집니다. 이 문서는 새로운 흐름과 그것을 받쳐줄 화면/레이아웃 옵션을 정리한 설계 탐색입니다.

01

핵심 발견

사고를 여는 한 줄
Key Insight

복합키 = (물류센터 + 발주번호 + SKU) 는 운송방법을 결정합니다. 한 복합키가 밀크런과 쉽먼트로 쪼개지는 일은 없습니다 — 항상 둘 중 하나입니다.

이 제약이 의미하는 것: 발주확정서 단계에서 운송방법만 정해두면, 그 다음은 두 갈래로 흩어뜨려서 처리할 수 있습니다. 같은 차수 안에서 쉽먼트 lot은 송장 들어오는 대로 먼저 올리고, 밀크런 lot은 팔레트가 정해지는 대로 따로 올리고. 사람도, 시점도, 페이스도 분리됩니다.

02

두 워크플로우 비교

현재 흐름 vs. 분리형 흐름

지금 (모놀리식)

현재

한 차수 = 한 묶음. 모든 단계가 끝나야 다음 차수로 넘어갈 수 있음. 송장 늦게 들어오는 쉽먼트 1건이 전체를 막는다.

1
PO 다운
쿠팡 사이트에서 SKU 목록 받기
2
투비 재고조정
협력사 재고 반영
3
확정서 임시 생성
셀러가 셀 단위로 보냄/반려/수량 결정
4
쉽/밀 운송 분배
팔레트·박스·송장 모두 입력
5
확정서 업로드 (위험)
쿠팡에 한 번에 push — 3초 카운트다운
6
운송분배 업로드 (위험)
사이트 운송지정 화면에 push
7
결과 다운로드
통합 양식 저장

제안 (분리형 / 인박스)

제안

확정 후 두 갈래(쉽/밀)로 흩어진 인박스에 쌓이고, 각 인박스가 자기 페이스로 마감. 차수의 "완료"는 두 인박스가 모두 빌 때.

1
반려 결정 경영지원
전체 PO에서 보냄 / 안 보냄. 수량은 default(=요청). 운송방법은 안 정해도 됨.
2
운송방법 지정 + 확정 물류
'보냄'으로 표시된 행에 운송방법(쉽/밀)만 지정 → 즉시 사이트에 발주확정 업로드
⚡
확정서 업로드 (위험)
사이트 push. 이 시점부터 두 인박스로 자동 라우팅됨.
3a
밀크런 인박스 처리 운송
대기중 밀크런 묶음 목록 → 팔레트/프리셋 추가 → 사이트 업로드 → 다운로드
3b
쉽먼트 인박스 처리 운송
대기중 쉽먼트 묶음 목록 → 박스 추가 + 송장 입력 → 사이트 업로드 → 다운로드
✓
결과 히스토리
두 인박스 모두 비면 차수 완료. 시점별 산출물 모두 보관.

제안 흐름의 본질: "차수"라는 컨테이너는 그대로 두되, 그 안의 작업을 두 종류의 인박스(쉽/밀)로 분리합니다. 차수는 더 이상 "선형 진행"이 아니라 "두 큐가 비어가는 과정"이 됩니다.

복합키 lifecycle (상태 머신)

한 row의 인생

복합키 row 1개는 다음 상태를 거칩니다. 각 상태에서 어떤 화면이 표시되고 어떤 작업이 가능한지가 명확해야 UI가 깔끔해집니다.

설계 함의:

  • METHOD_NEEDED 게이트가 새로 생김 — 기존엔 운송분배 화면에서 암묵적으로 끝났던 것을 명시적 단계로
  • IN_INBOX 이후는 두 큐가 독립 진행. 차수 진행률은 두 큐의 잔여 합으로 계산
  • 각 ⚡(위험 전이)에 카운트다운 + 취소. 그 외는 일반 저장
  • 되돌리기 가능한 전이 ↔ 불가능한 전이를 시각적으로 구분 (위로 가는 화살표는 별도 색)
05

운송 인박스 모델

두 갈래 큐의 작동 방식

발주확정 직후, 자동 라우팅

'보냄'으로 확정된 복합키 row가 운송방법에 따라 두 인박스에 자동 떨어집니다. 차수가 다른 lot이 같은 인박스에 섞여 있어도 OK — 인박스는 차수가 아니라 운송방법으로 묶입니다.

06

레이아웃 옵션

웹뷰·로그 처리 방식 4가지

현재는 우측 슬라이드 패널 안에 웹뷰. 하지만 BrowserView/WCV는 native overlay라 다른 옵션이 가능합니다. 작업 로그도 마찬가지로 분리 가능. 4가지 안:

A · 우측 슬라이드 (현재)
웹뷰는 우측 패널, 로그는 메인 하단. 모두 한 윈도우.
+ 한 시야에 모든 게 있음
− 메인 작업 영역이 좁음 · 송장 입력 같은 키보드 헤비 작업이 답답
웹뷰 창
로그 창
B · 분리 윈도우 추천
메인은 작업에만 집중. 웹뷰 / 로그는 별도 윈도우로 띄움. 멀티 모니터 사용자에게 최적.
+ 메인 작업 공간 풀폭 · 모니터 분리 자연스러움
− 단일 모니터에서는 알트탭이 늘어남 · 항상 위 옵션 필요
웹뷰 — 자동화 동작 시 모달로 점프
C · 자동화 모달
평소 웹뷰는 숨김. 위험 동작(업로드) 트리거 시 화면 가운데 큰 모달로 떠서 사용자가 검수. 끝나면 자동 닫힘.
+ 평소 메인 작업 풀폭 · 위험 동작에만 집중하게
− 모달이 자주 뜨면 피로 · 자동화 동작 길면 답답
D · 하단 드로어
웹뷰 + 로그를 통합한 하단 패널. 탭으로 둘 사이 전환. VS Code 터미널 영역과 같은 패턴.
+ 데스크톱 앱 사용자에게 친숙 · 와이드 모니터 잘 활용
− 폭이 좁은 모니터에서는 둘 다 답답 · 가로 그리드 중심인 우리 작업과 충돌

제안: 옵션 B(분리 윈도우)를 기본으로, 옵션 C(모달)를 위험 동작 시에만 자동 투입. 옵션 A는 단일 모니터/노트북 사용자를 위한 폴백 모드. 사용자가 '환경'을 한 번 선택하면 그 모드가 유지되게.

07

화면 인벤토리 (재구성)

제안 흐름 기준 화면 목록
#화면역할핵심 기능새로움
S0대시보드경영지원오늘 처리해야 할 차수 + 두 인박스 잔여 + 어제 마감 안 된 lot신규
S1달력전체월 단위 / 차수 카드 + 진행률 점유지
S2차수 검토 (PO)경영지원그리드. 행별 보냄/반려 토글, 부족사유. 운송방법 칸은 회색 (다음 단계)단순화
S3운송방법 + 확정물류'보냄' 행만 보임. 운송방법 일괄/개별 지정. 하단에 "확정 업로드" 큰 버튼신규
S4a밀크런 인박스운송대기중 lot 카드 리스트. 카드 클릭 → S5a신규
S4b쉽먼트 인박스운송대기중 lot 카드 리스트. 카드 클릭 → S5b신규
S5a밀크런 lot 편집운송팔레트 추가 + SKU×팔레트 매트릭스. 끝나면 업로드분리 (모달 가능)
S5b쉽먼트 lot 편집운송박스 추가 + 송장 + SKU×박스 매트릭스. 끝나면 업로드분리 (모달 가능)
S6결과 히스토리전체차수별 산출물 다운로드 + lot별 업로드 결과유지
S7설정 / 플러그인전체벤더, 자격증명, 마스터, 라이선스유지

화면 수가 늘어난 것처럼 보이지만, 실제로는 한 사람이 한 번에 보는 정보는 줄어듭니다. 경영지원은 S0+S2만 보면 됨. 물류는 S3만. 운송은 S4a/S4b만. 한 사람이 다 한다면 좌측 사이드바 순서대로 따라가면 됨.

08

열린 질문

결정해야 진행 가능

인박스의 단위는?

벤더 단위로 인박스를 분리할지, 전 벤더 통합 단일 인박스로 갈지. 사용자가 한 번에 한 벤더만 보는 일반적이라면 벤더별, 통합 운영팀이라면 통합.

"차수"는 살아남는가?

인박스 모델로 가면 "차수"는 메타데이터일 뿐. 캘린더의 차수 카드는 무엇을 의미해야 하는가 — 발주확정의 단위? 또는 단순 그룹핑?

역할 기반 권한이 필요한가?

경영지원이 운송 화면을 못 보게 막을 필요가 있나, 아니면 그냥 사이드바 순서로 가이드만? 사용자 수가 적으면 후자.

위험 동작 카운트다운 위치

3개 업로드 지점(확정·밀크런·쉽먼트) 모두 카운트다운? 첫 업로드만? 위험도가 다르므로 차등 가능.

실패 시 롤백 정책

밀크런 업로드 후 쉽먼트 업로드가 실패하면? 차수 단위 atomic이 아니라 lot 단위 atomic. 부분 실패가 일상.

WMS 결과 도착 시점

WMS 결과는 발주확정 후에 들어옴. 그렇다면 S3 직후 잠시 "WMS 대기" 상태가 있어야 함. 자동 폴링? 사용자가 수동 업로드?
