# WMS → 우리 앱 통합 엑셀 v2 스펙

WMS 가 PO 의 `물류 작업 계획` 시트를 받아 작업한 결과를 우리 앱에 돌려줄 때 사용하는 단일 통합 엑셀 양식.

업로드 한 번으로:
- PO 의 `확정수량` 갱신
- `재고조정` 의 확정수량 갱신
- `운송분배` (transport.json) 완전 채움 (lots / pallets / boxInvoices / items)

샘플: [wms-output-v2-sample.xlsx](./wms-output-v2-sample.xlsx)

---

## 시트 구조

### 시트 1: `확정`
WMS 가 **창고에서 추가로 출고 가능한 수량** 만 적음. 반출수량(이미 풀필로 빠진 양)은 우리 앱이 관리하므로 WMS 는 신경 X.

최종 확정수량은 우리 앱이 계산:
```
최종확정수량 = 반출수량(po-tbnws) + WMS확정수량(시트1)
```
이 값으로 `confirmation.xlsx` / `po-tbnws.xlsx` / `재고조정` 의 확정수량 컬럼 갱신.

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| 물류센터 | text | ✅ | PO 의 물류센터명과 정확히 일치 |
| 발주번호 | text | ✅ | PO 의 발주번호 |
| 상품코드 | text | – | 참고용 (G-9995-0001 등) |
| SKU Barcode | text | ✅ | 매칭 키 |
| 상품명 | text | – | 참고용 |
| 확정수량 | int | ✅ | 창고 출고 가능 수량 (반출 제외). 0 이면 추가 출고 없음 |

매칭 키: `(물류센터, 발주번호, SKU Barcode)`

### 시트 2: `컨테이너`
박스 묶음(쉽먼트) 또는 팔레트(밀크런) 1개당 1행.

| 컬럼 | 타입 | 필수 (쉽먼트) | 필수 (밀크런) | 설명 |
|---|---|---|---|---|
| 물류센터 | text | ✅ | ✅ | |
| 컨테이너ID | text | ✅ | ✅ | 자유 식별자. 시트3 와 매칭 (`S-1`, `P-1`, `P-2` …) |
| 운송방법 | enum | ✅ | ✅ | `쉽먼트` 또는 `밀크런` |
| 박스수 | int | ✅ | ✅ | 그 컨테이너 안 박스 수 |
| 송장번호 | text | ✅ | – | 쉽먼트 송장번호 |
| 출고지 | text | – | ✅ | 마스터 시트의 출고지 **이름** 그대로 입력 (예: "지티창고"). 우리 앱이 ID 로 변환 |
| 프리셋 | text | – | ✅ | 마스터 시트의 프리셋 **이름** 그대로 (예: "아주팔레트 대") |
| 비고 | text | – | – | 자유 텍스트 |

### 시트 3: `적재`
SKU × 컨테이너 매핑. 각 SKU 가 어느 컨테이너에 몇 개 들어갔는지.

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| 물류센터 | text | ✅ | |
| 발주번호 | text | ✅ | |
| SKU Barcode | text | ✅ | |
| 컨테이너ID | text | ✅ | 시트2 의 컨테이너ID 와 정확히 일치 |
| 수량 | int | ✅ | 그 컨테이너에 적재된 그 SKU 의 수량 |

### 시트 4: `마스터` (read-only)
우리 앱이 PO 보낼 때 자동 채움. WMS 가 시트2 의 출고지/프리셋 입력 시 **정확한 이름** 을 참고할 수 있게 노출.

- **출고지** 섹션: ID / 이름 / 주소
- **프리셋** 섹션: 이름 / 가로 / 세로 / 높이 / 렌탈사
- **운송방법 enum** 섹션: `쉽먼트` / `밀크런`

WMS 는 이 시트를 보고 시트2 의 출고지/프리셋 컬럼에 **이름** 만 정확히 복붙하면 됨.

## 검증 규칙 (apply 시 강제)

### 참조 무결성
1. 시트1 의 `(물류센터, 발주번호, Barcode)` 가 PO 에 존재해야 함
2. 시트3 의 `(물류센터, 발주번호, Barcode)` 가 시트1 에 존재해야 함
3. 시트3 의 `(물류센터, 컨테이너ID)` 가 시트2 에 존재해야 함

### 수량 정합성
4. SKU별 SUM(시트3.수량) = 시트1.확정수량 (= WMS 가 출고할 양만, 반출분 제외)
5. 컨테이너별 SUM(시트3.수량) ≤ 시트2.박스수 × 1박스당평균(soft, 경고만)
6. 시트2.박스수 = 그 컨테이너에 실제 들어간 SKU 박스 수 합. 검증은 `SUM(시트3.수량) ≤ 시트2.박스수` (수량 단위가 박스가 아닐 수도 있어 합과 박스수 직접 매칭은 X)
7. 최종확정수량(반출+WMS) ≤ 신청수량 — 초과 시 경고 (우리 앱 계산 후 검증)

위반 시: 시트1/2/3 별 누락·불일치 행을 모달로 표시하고 사용자가 무시하거나 취소 결정.

## 우리 앱이 이걸 읽어 처리하는 흐름

1. **시트1 → PO/재고조정 갱신**
   - `po.xlsx`, `po-tbnws.xlsx` 의 `(센터, 발주, 바코드)` 매칭 행의 `확정수량` 컬럼 patch
   - `coupang-export.json` 의 SKU 스냅샷 갱신
   - `confirmation.xlsx` 의 SKU 행 갱신 (확정수량 0 → 납품제외, 부족사유 자동 채움)
2. **시트2 → transport.json `assignments[wh].lots[]` 생성**
   - 운송방법별로 lot 묶기 (한 센터에 쉽먼트 lot + 밀크런 lot 공존 가능)
   - 쉽먼트 lot: `boxCount = SUM(쉽먼트 컨테이너 박스수)`, `boxInvoices[i] = 송장번호`
   - 밀크런 lot: `originId = 출고지`, `pallets[].presetName/boxCount` 채움, `totalBoxes = SUM(밀크런 컨테이너 박스수)`
3. **시트3 → lot.items[] 생성**
   - 컨테이너ID 의 운송방법으로 분기 → 쉽먼트 lot.items 에 `{rowKey, qty, boxNo}`, 밀크런 lot.items 에 `{rowKey, qty, palletNo}`
   - boxNo/palletNo = 시트2 의 컨테이너ID 순서 (1-based)
4. **confirmation.xlsx 의 입고유형 patch**
   - 시트3 매핑으로 SKU 별 운송방법 결정 → `입고유형` 컬럼 갱신

업로드 후 **운송분배 모달 자동 오픈** — 사용자가 검토만 (수정 불필요).

## WMS 측 구현 가이드 (요약)

WMS 가 입력으로 받는 PO 의 `물류 작업 계획` 시트를 보고:

1. **확정수량 결정**: 출고 가능 수량 만큼 → 시트1 채움 (0 면 납품 제외)
2. **컨테이너 결정**: 박스/팔레트 단위로 어떻게 출고할지 → 시트2 채움
3. **적재 매핑**: SKU 별로 어느 컨테이너에 몇 개 → 시트3 채움

검증 규칙 4 (`SUM(시트3) = 시트1.확정수량`) 만 지키면 round-trip 보장됨.

## 미해결 / 추후 결정

- **출고지 ID 체계**: WMS 가 쿠팡 출고지 seq 를 정확히 알기 어려우면, 출고지명(예: "지티창고") 으로 두고 우리 앱에서 settings 룩업 fallback 추가
- **혼합 입고유형 SKU**: 한 SKU 가 쉽먼트 + 밀크런 둘 다 걸치면 confirmation.xlsx 의 입고유형은 첫 컨테이너의 운송방법으로 결정 (현재 정책 유지)
- **컨테이너ID 자유도**: 자유 입력이라 오타 위험. WMS 측에서 자동 생성(예: `${운송방법}-${row}`) 권장
